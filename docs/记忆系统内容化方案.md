# 记忆系统内容化方案

## 1. 文档目标

本文件只负责整理“已有系统能力如何沉淀成可复用内容资产”，不再混入架构规范和开发排期。

- 当前基线日期：2026-04-15
- 适用范围：`Memory Box / Memory-System` 主工程

## 2. 当前判断

系统当前不缺功能点，缺的是把已有能力整理成稳定的内容入口、展示模板和证据索引。

现有内容原料：

1. 四层记忆架构与“文件即记忆”方法
2. Memory Reflection 与 Evidence Trace
3. 摘要、时间线、复盘、治理与报告视图
4. Skill、RAG、任务、主动提醒、多端统一身份
5. benchmark 与评测结果归档

当前短板：

1. 内容入口分散
2. 演示脚本还不够固定
3. benchmark 结果尚未形成统一摘要模板
4. 治理案例与失败案例缺少持续沉淀方式

## 3. 内容主线

建议长期只围绕五条主线组织内容：

1. 系统设计线：四层架构、文件即记忆、模块分层、启动导航
2. 运行机制线：记忆如何提取、检索、注入、落盘
3. 治理与可信线：冲突审批、证据链、可解释与可审计
4. 主动服务线：任务、提醒、周报、跨端连续性
5. 评测与实验线：benchmark、对照结果、失败样例、修复前后变化

## 4. 证据入口映射

| 内容主题 | 首选入口 | 说明 |
| --- | --- | --- |
| 架构拆解 | `README.md`、`开发文档.md` | 讲系统设计与边界 |
| 单轮证据解释 | `/memory-debug [N]` | 展示 reflection 与 evidence trace |
| 记忆总览 | `/memory-report` | 展示当前状态与统计摘要 |
| 记忆复盘 | `/memory-review` | 串联摘要、治理、任务和证据 |
| 治理案例 | `/memory-governance` | 展示冲突、待审核与处理结果 |
| 时间演化 | `/memory-timeline`、`/memory-scenes` | 展示长期记忆形成过程 |
| 主动服务 | `/tasks`、`/weekly-report`、`/proactive-reminders` | 展示任务闭环与主动提醒 |
| 评测结果 | `/benchmark`、`/benchmark history [N]`、`/benchmark reports [N]`、`eval_results.jsonl` | 形成评测与趋势对比 |

## 5. 推荐内容资产

优先沉淀以下资产：

1. 架构拆解稿：对应四层架构、文件即记忆与模块边界
2. 固定演示脚本：至少 3 个稳定场景
3. benchmark 结果摘要模板：统一批次结论和指标口径
4. 治理案例模板：记录冲突、审批、结果与解释
5. 失败样例模板：记录问题、原因、修复与回归方式

## 6. 建议目录

建议逐步把内容资产固定到以下目录：

1. `docs/`：长期稳定文档
2. `.memory/content-assets/demo/`：演示脚本与场景卡片
3. `.memory/content-assets/benchmark/`：评测摘要、对照结果、图表素材
4. `.memory/content-assets/cases/`：治理案例与失败样例

## 7. 近期执行建议

1. 固化 3 个稳定演示场景，统一命令顺序与预期输出。
2. 跑通一轮 benchmark，并沉淀结果摘要模板。
3. 选 1 到 2 个治理案例，形成可复用案例模板。
4. 建立失败样例记录方式，方便后续回归与展示。

## 8. 维护约定

1. 新增内容资产时，优先在本文件补入口与模板说明。
2. 不再把内容规划写回 `开发文档.md`。
3. 每项内容都应回指到具体命令、文件或评测结果，避免空泛描述。
