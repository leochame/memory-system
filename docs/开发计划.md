# 开发计划

## 1. 文档目标

本文件用于统一维护项目当前阶段的后续开发计划、优先级、验收标准和收尾口径。

- 当前基线日期：2026-04-08
- 当前项目阶段：主体功能完成，进入收尾与答辩增强阶段

---

## 2. 当前阶段目标

当前阶段的目标不是继续堆叠大而全的新功能，而是围绕以下四类工作完成收口：

1. 固化真实模型/API 的端到端可复现基线
2. 建立评测、消融实验与论文图表所需的数据闭环
3. 清理尚未定案的历史兼容结构与预留文件
4. 补齐少量高价值答辩增强能力

---

## 3. 优先级计划

### P0 - 必须优先完成

#### 3.1 真实模型/API E2E 基线固化

- 目标：
  - 在非临时模式下跑通真实模型/API 的主链路
  - 固化一版可复现的基线结果
  - 为答辩演示和论文实验提供“真实可运行”证据
- 交付：
  - 至少一次成功的 `./scripts/run-real-api-e2e.sh` 结果记录
  - 基线日志归档到 `logs/e2e/`
  - 在文档中补充执行环境、模型、样例输入和结果摘要
- 验收标准：
  - 主链路能稳定完成对话、记忆写入、工具调用或检索
  - E2E 结果可被重复执行并得到同口径输出

#### 3.2 评测指标与实验闭环

- 目标：
  - 建立面向论文的最小实验体系
  - 输出可比较的指标与对照结果
- 当前状态：
  - 已完成基础自动化入口：`/benchmark`、`/benchmark history [N]`
  - 已支持单题 A/B 评测与内置评测集批量运行
  - 下一步重点转为“扩数据集、补专项指标、做消融”
- 交付：
  - 评测样例集
  - 核心指标定义
  - 消融实验方案
  - 论文图表所需数据口径
- 建议指标：
  - 显式记忆提取准确率
  - 隐式洞察提取质量
  - RAG 召回命中率
  - 记忆引用命中率
  - 响应延迟
  - 摘要压缩收益
- 验收标准：
  - 至少能跑通一套固定评测流程
  - 至少完成一版对照实验结果沉淀

### P2 - 答辩增强能力

#### 3.5 任务闭环看板

- 目标：补齐“创建 -> 提醒 -> 完成 -> 复盘”的演示闭环
- 建议交付：
  - `/tasks today`
  - `/tasks overdue`
  - `/tasks done`
  - 任务完成、延期、忽略操作
- 验收标准：
  - 能在 CLI 中稳定展示任务状态流转

#### 3.6 记忆审批命令

- 目标：把记忆治理从“可查看”升级为“可操作”
- 当前状态：已完成基础 CLI 版本（2026-04-08）
- 建议交付：
  - `/memory-approve <序号>`
  - `/memory-reject <序号>`
- 验收标准：
  - 冲突记忆与待审核记忆可以完成显式处理

#### 3.7 飞书卡片化

- 目标：提升产品化展示效果
- 建议交付：
  - 周报卡片
  - 任务提醒卡片
  - 完成/延期/忽略按钮
- 验收标准：
  - 至少一种高频消息场景能以卡片形式稳定发送

#### 3.8 一键答辩演示模式

- 目标：降低现场演示出错概率
- 建议交付：
  - `/demo run`
  - 串联无记忆、个人记忆、团队记忆、周报推送等固定脚本
- 验收标准：
  - 能按固定顺序稳定复现关键演示片段

---

## 4. 建议执行顺序

1. 先完成真实 API E2E 基线固化
2. 同步建立最小评测样例和指标口径
3. 再处理 `model_set_context.json` 与 `memory_queues.json` 的去留决策
4. 最后补做答辩增强能力

原因：

- 前两项直接决定论文数据和系统可信度
- 中间两项影响代码与文档口径统一
- 最后一组更偏展示增强，价值高但不应阻塞基线收口

---

## 5. 完成定义

当以下条件满足时，可认为项目进入“答辩可封版”状态：

1. 真实 API E2E 基线已完成并有文档记录
2. 评测指标、样例、至少一版对照结果已沉淀
3. 历史兼容文件与预留文件的去留已明确
4. 至少完成 1 到 2 个高价值答辩增强项
5. `README.md`、`开发文档.md`、`开发实现process.md`、本文件保持一致
