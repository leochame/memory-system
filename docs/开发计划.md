# 开发计划

## 1. 文档目标

本文件用于统一维护项目当前阶段的后续开发计划、优先级、验收标准和收口顺序。

- 当前基线日期：2026-04-15
- 当前阶段：主体功能完成，进入收尾、实验与展示增强阶段

## 2. 当前阶段目标

当前阶段不再继续扩张大而全的新功能，而是围绕以下事项完成收口：

1. 固化真实模型/API 的端到端可复现基线
2. 建立 benchmark、指标与实验数据闭环
3. 清理兼容结构与预留文件的口径歧义
4. 补少量高价值答辩演示能力
5. 保持项目文档职责边界清晰且同步

## 3. 优先级计划

### P0 - 必须优先完成

#### 3.1 真实模型/API E2E 基线固化

- 目标：
  - 在非临时模式下跑通真实模型/API 主链路
  - 固化一版可复现的基线记录
  - 为答辩演示和实验提供真实运行证据
- 交付：
  - 至少一次成功的 `./scripts/run-real-api-e2e.sh` 结果记录
  - 基线日志归档到 `logs/e2e/`
  - 文档补充执行环境、模型、样例输入和结果摘要
- 验收标准：
  - 主链路能稳定完成对话、记忆写入、工具调用或检索
  - 相同流程可重复执行并得到同口径结果

#### 3.2 评测指标与实验闭环

- 目标：
  - 把现有 benchmark 从“能跑”升级为“可比较、可复用、可归档”
- 当前状态：
  - 已支持 `/benchmark`
  - 已支持 `/benchmark history [N]`
  - 已支持 `/benchmark reports [N]`
  - 已支持 `/benchmark dataset`
- 下一步交付：
  - 扩充外部题集
  - 定义核心指标口径
  - 补齐消融实验方案
  - 固化图表和汇总模板
- 建议指标：
  - 显式记忆提取准确率
  - 隐式洞察提取质量
  - RAG 召回命中率
  - 记忆引用命中率
  - 响应延迟
  - 摘要压缩收益
- 验收标准：
  - 至少能跑通一套固定评测流程
  - 至少沉淀一版可复用的对照实验结果

### P1 - 需要尽快定案

#### 3.3 `model_set_context.json` 去留决策

- 现状：文件存在，但未进入 runtime 主链路
- 待决策方向：
  - 正式接入并明确职责
  - 保留为预留结构并在文档中明确“不参与主流程”
  - 确认废弃并移除
- 验收标准：
  - 代码、存储、文档口径一致

#### 3.4 `memory_queues.json` 去留决策

- 现状：单文档画像已稳定，但队列结构仍有运行时兼容用途
- 待决策方向：
  - 继续保留，并明确只负责队列/兼容信息
  - 提供迁移或替代方案后正式退役
- 验收标准：
  - 用户画像主链路不存在“双数据源”歧义

### P2 - 展示增强能力

#### 3.5 任务闭环看板

- 目标：补齐“创建 -> 提醒 -> 完成 -> 复盘”的展示闭环
- 建议交付：
  - `/tasks today`
  - `/tasks overdue`
  - `/tasks done`
  - 任务完成、延期、忽略操作
- 验收标准：
  - CLI 中能稳定展示任务状态流转

#### 3.6 飞书卡片化

- 目标：提升主动提醒和周报的展示效果
- 建议交付：
  - 周报卡片
  - 任务提醒卡片
  - 完成/延期/忽略按钮
- 验收标准：
  - 至少一种高频消息场景支持卡片化稳定发送

#### 3.7 一键 demo 模式

- 目标：降低现场演示操作成本和出错率
- 建议交付：
  - `/demo run`
  - 固定场景脚本串联
- 验收标准：
  - 能稳定复现关键演示片段

### P3 - 文档与资产收口

#### 3.8 项目文档持续收口

- 目标：
  - 保持 `README.md`、`开发文档.md`、`开发实现process.md`、本文件口径一致
- 验收标准：
  - 总览、架构、进度、计划四类信息不再重复堆叠

#### 3.9 内容资产固化

- 目标：
  - 把已有命令、案例、benchmark 结果沉淀为可复用展示资产
- 建议交付：
  - 稳定场景脚本
  - benchmark 结果摘要模板
  - 治理案例模板
- 验收标准：
  - 每份资产都能回指到具体命令、文件或评测结果

## 4. 建议执行顺序

1. 先固化真实 API E2E 基线
2. 再补 benchmark 数据集与指标口径
3. 然后明确 `model_set_context.json` 与 `memory_queues.json` 的去留
4. 最后补展示增强项与内容资产

## 5. 完成定义

当以下条件满足时，可认为项目进入“答辩可封版”状态：

1. 真实 API E2E 基线已完成并有文档记录
2. benchmark 数据、指标、至少一版对照结果已沉淀
3. 历史兼容文件与预留文件去留已明确
4. 至少完成 1 到 2 个高价值展示增强项
5. 项目文档保持一致且职责边界清晰
