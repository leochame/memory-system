# 毕业论文工作流总说明

本目录服务于 `Memory-System / Memory Box` 的**毕业论文写作**，不是项目开发工作流。

这里的所有角色都必须明确一件事：

- 当前任务是在写毕业论文
- 参考对象是当前仓库里的真实项目实现
- 目标是把“已经实现的系统”写清楚、写真实、写得像毕业论文
- 不是继续开发功能，不是补代码，不是虚构实验结果

## 1. 工作流目标

本目录的作用只有三件事：

1. 由作者整理写作计划与资料
2. 直接修改论文正文
3. 对论文全文进行审阅并给出 review 意见

所有角色都必须围绕“论文写作”工作，而不是“项目开发”工作。

## 2. 角色与输出位置

### Student Author

- Prompt 文件：`毕设文档/workflow/prompts/student-author.md`
- 维护文件：`毕设文档/workflow/01_plan.md`
- 正文文件：`毕设文档/毕业论文初稿.md`
- 作用：负责资料收集、写作计划整理、直接正文撰写，并主动控制 AI 痕迹与重复风险

### Thesis Reviewer

- Prompt 文件：`毕设文档/workflow/prompts/thesis-reviewer.md`
- 作用：以论文全文为主检查真实性、论文口吻、AI 痕迹、格式与重复风险，并直接提供 review 意见

## 3. 统一参考来源

所有角色都必须优先参考以下文件，而不是凭常识自由发挥：

### 项目实现参考

- `开发实现process.md`
- `docs/开发计划.md`
- `开发文档.md`
- `README.md`
- 当前仓库相关源码文件
- 当前仓库相关测试文件

### 论文参考

- `毕设文档/毕业论文初稿.md`
- `毕设文档/workflow/01_plan.md`
- `毕设文档/workflow/AI痕迹检查清单.md`

原则：

- 写论文时必须参考项目实现
- 作者整理计划和资料时必须参考当前论文状态
- 审稿时必须同时参考论文初稿、计划和项目实现

## 4. 统一禁止事项

所有角色统一禁止以下行为：

1. 把路线图、设计方案、计划项写成已实现成果
2. 把项目开发建议写进论文正文
3. 虚构实验结果、测试数据、运行表现或模块职责
4. 把“后续工作”写成“当前版本已经完成”
5. 以产品宣传、答辩话术或泛化 AI 说明替代论文写作
6. 忘记当前身份是在写毕业论文，而不是在继续开发项目
7. 把“降低 AI 率”理解成只靠 reviewer 或工具兜底，而作者自己不主动修改

## 5. 文件职责边界

- `01_plan.md`
  - 作者维护的写作计划、资料清单、修改范围
- `毕业论文初稿.md`
  - 作者直接修改的论文正文

要求：

- 不把开发过程流水账写进 `毕业论文初稿.md`
- 作者可以直接修改 `毕业论文初稿.md`
- reviewer 直接对 `毕业论文初稿.md` 全文 review，并在 review 时提供意见
- 作者如需整理资料来源、修改边界和写作提醒，应优先放在 `01_plan.md`
- reviewer 在检查 AI 痕迹与重复风险时，应参考 `AI痕迹检查清单.md`

## 5.5 两角色分工

### Student Author

负责：

1. 收集与当前章节相关的项目资料
2. 维护 `01_plan.md` 中的写作计划、资料依据和修改范围
3. 直接修改 `毕业论文初稿.md`
4. 在写作阶段主动检查 AI 痕迹、重复表达和模板化句式

不负责：

- 审核全文质量结论
- 判断查重/AI 痕迹风险是否可接受

### Thesis Reviewer

负责：

1. 对 `毕业论文初稿.md` 全文进行 review
2. 检查格式、表述、重复、AI 痕迹和真实性风险
3. 直接提供 review 意见

不负责：

- 代替作者大规模收集资料
- 代替作者撰写正文初稿

## 6. 当前阶段建议

结合当前项目状态，论文阶段建议优先处理：

1. 第 6 章测试、benchmark、实验指标与结果分析
2. 第 5 章实现细节与实现映射边界
3. 第 7 章总结、局限与展望
4. 摘要和第 1 章整体语气统一

这意味着：

- 当前 workflow 的主要任务是补强论文证据链
- 不是再去扩展新的开发需求
- 所有角色都要帮助论文更真实、更稳、更可答辩
