# Memory Box 毕设展示脚本

> 总时长建议 15-20 分钟，按以下顺序操作即可。

## 开场（1 分钟）

**你说的话：**

> 老师好，我的毕设题目是"四层智能记忆管理系统"，用 Java + Spring Boot + LangChain4j 实现。
> 核心解决的问题是：**对话系统如何跨会话记住用户信息，并在合适的时机使用这些记忆。**
> 我直接用系统演示。

## 第一步：启动系统（1 分钟）

打开终端，进入项目目录：

```bash
cd /Users/leocham/Documents/code/memory-system
java -jar target/memory-box-1.0.0.jar
```

启动后你会看到欢迎界面，显示 `mode=memory · rag=on`。

**你说的话：**

> 系统启动后自动加载记忆模块，RAG 默认开启。下面我先展示记忆是怎么被记录和使用的。

---

## 第二步：展示用户画像（2 分钟）

输入命令：

```
/what-you-know
```

**你会看到**：系统输出完整用户画像（姓名、专业、毕设项目、技术栈、IDE、学习兴趣、编程习惯、饮食偏好等 9 条记忆）。

**你说的话：**

> 这是 Layer 3 用户洞察。系统通过多轮对话，把用户的长期信息提取出来，持久化到本地文件。这些记忆不是每轮都全量加载的，而是通过记忆反思机制按需读取。

---

## 第三步：展示记忆反思与证据链（3 分钟）

输入命令：

```
/memory-debug 10
```

**你会看到**：最近 10 条证据追踪记录，每条都包含：
- 用户的输入
- 反思判断：是否需要加载记忆（`needs_memory: true/false`）
- 判断理由
- 实际加载了哪些记忆
- 最终使用了哪些记忆

**你说的话：**

> 这是记忆反思机制的核心。每次用户发消息时，系统先判断这条消息需不需要历史记忆。
> 比如"你好"这种简单问候，系统判断不需要加载记忆，直接回复；
> 但"你能记住我之前说过什么吗？"这种，系统会加载全部画像来回答。
> 每一轮的记忆使用都会留下证据追踪，保证可解释、可审计。

---

## 第四步：展示四层架构全景（2 分钟）

输入命令：

```
/memory-report
```

**你会看到**：系统综合状态报告，包括各层记忆的数量、最后更新时间、RAG 索引状态等。

**你说的话：**

> 四层记忆架构：
> - Layer 1 短期记忆：最近对话窗口
> - Layer 2 元数据：用户环境、偏好、全局控制
> - Layer 3 用户洞察：长期画像（刚才看到的）
> - Layer 4 持续学习知识：包括 Skill（方法论按需加载）和 RAG（语义检索历史案例）
>
> 每一层独立存储、按需读取，不会把所有东西塞进同一个上下文。

---

## 第五步：展示 Skill 按需加载（2 分钟）

输入命令：

```
/skills
```

**你会看到**：3 个技能文件列表（RAG 检索指南、四层架构说明、调试方法论）。

**你说的话：**

> Skill 是 Layer 4a，本质是结构化的方法论文档。系统不会一次性把所有 Skill 内容注入 prompt，而是先暴露可用名称，只有当反思结果判断需要时，才通过 `load_skill(name)` 工具读取单个 Skill 的正文。这是工具优先的渐进式加载策略。

---

## 第六步：展示会话摘要与时间线（2 分钟）

输入命令：

```
/memory-timeline
```

**你会看到**：按时间排列的会话摘要、话题切换、里程碑记录。

**你说的话：**

> 系统在两种条件下自动生成摘要：对话达到轮次阈值，或者检测到话题切换。
> 这些摘要一方面用于上下文压缩，另一方面可以展示对话的时间线和关键事件。

---

## 第七步：展示记忆治理（2 分钟）

输入命令：

```
/memory-governance
```

**你会看到**：2 条冲突记忆待审核（比如"编程语言偏好"和"作息习惯"出现了新旧冲突）。

**你说的话：**

> 记忆不是只增不改的。当用户说了和之前记忆矛盾的内容，系统不会直接覆盖，而是把冲突放入待审核队列。用户可以通过 `/memory-approve` 接受新记忆，或 `/memory-reject` 保留旧记忆。这是记忆的治理机制。

---

## 第八步：展示任务管理与主动服务（2 分钟）

输入命令：

```
/tasks
```

**你会看到**：4 个任务，2 个已触发（中期检查、组会），2 个待触发（找导师讨论、Rust 练习）。

然后输入：

```
/proactive-reminders
```

**你会看到**：2 条系统主动生成的提醒（学习建议、效率提示）。

**你说的话：**

> 系统不仅能被动响应，还能基于用户画像主动生成建议。比如根据"喜欢晚上写代码"的习惯，系统会主动提醒利用晚间时段做高效工作。

---

## 第九步：展示 Benchmark 评测（3 分钟）

输入命令：

```
/benchmark reports 5
```

**你会看到**：一份批次评测报告，包含 3 道题的 A/B 对比（有记忆 vs 无记忆），4 个维度的评分，以及提升百分比。

然后输入：

```
/benchmark dataset
```

**你会看到**：当前评测题集（10 道题）。

**你说的话：**

> 这是评测系统的核心能力。每道题会在两种模式下分别回答：
> - 无记忆模式：关闭所有记忆，像普通 ChatBot 一样回复
> - 有记忆模式：完整加载四层记忆后回复
>
> 然后 LLM 从相关性、个性化、准确度、有用性四个维度打分。
> 从结果看，有记忆模式在个性化维度提升最明显，最高的题目提升了 184%。
> 这也是论文里实验章节的数据来源。

---

## 第十步：展示多端统一身份（1 分钟）

输入命令：

```
/identity
```

**你会看到**：3 个身份映射（CLI、飞书、Telegram），都关联到同一个用户。

**你说的话：**

> 系统支持 CLI、飞书、Telegram 三端接入，通过统一身份映射，用户在不同端的记忆是打通的。

---

## 收尾（1 分钟）

输入命令：

```
/rag-stats
```

**你会看到**：向量索引统计。

**你说的话：**

> 以上就是系统的完整展示。总结一下，我的毕设核心贡献：
> 1. 设计了四层记忆架构，覆盖从短期到长期、从被动到主动的记忆管理
> 2. 实现了记忆反思机制，避免无意义地全量加载记忆
> 3. 建立了证据追踪，让记忆使用可解释、可审计
> 4. 提供了 A/B 对比的评测框架，能量化记忆带来的提升
>
> 代码大约 8000+ 行，全部在单仓库内，可以通过 `mvn compile` 验证编译。

## 注意事项

1. **展示前确认已运行** `python3 scripts/demo-data-generator.py` 生成演示数据
2. **展示前先编译** `./scripts/run-compile.sh` 确保 jar 包是最新的
3. 如果遇到 API 不可用（无网络），系统启动会报错但不影响 CLI 命令查看已有数据
4. `/weekly-report` 需要调用 LLM，如果 API 不可用会报错，其他查看类命令不受影响
5. 展示过程中老师随时可能打断提问，脚本顺序可以灵活调整
