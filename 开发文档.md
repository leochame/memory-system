# 记忆管理系统 - 开发文档

## 1. 项目概述

### 1.1 项目目标
构建一个多层级、自动化管理且用户高度可控的记忆管理系统，参考 ChatGPT 的记忆机制，实现个性化的对话体验。

### 1.2 核心功能
- 多层级系统提示词架构（6个核心区块）
- 显式与隐式记忆提取引擎
- 自动记忆管理与 Top of Mind 算法
- 用户控制面板与隐私保护
- 文件系统持久化存储

## 2. 技术栈与架构决策

### 2.1 技术栈选择（已更新）

**最终决策：**
1. **编程语言**：Java 17+
   - **说明**：本次实现统一采用 **Java** 技术栈。
   - **选择 Java + Spring Boot 的理由**：
     - Spring Boot 在工程化、配置管理和依赖注入方面成熟稳定
     - 与 LangChain4j 生态良好集成，便于接入各类 OpenAI 格式模型
     - 适合未来扩展为长生命周期服务（即便当前仍以 CLI 交互为主）
2. **大模型集成**：使用 OpenAI 格式 API（兼容市面上大多数模型，包括 OpenAI、Claude、本地模型等），通过 **LangChain4j** 进行交互
3. **用户界面**：**纯 CLI 交互**，通过 Spring Boot 应用中的 `CommandLineRunner` / 自定义 CLI 入口实现（无 Web 控制器）
4. **依赖管理**：使用 Maven（或 Gradle），通过 `pom.xml`（或 `build.gradle`）管理依赖
5. **后台任务**：使用 Spring 的 `@Scheduled` 定时任务（或 `ScheduledExecutorService` 等 Java 并发工具）处理后台定时任务（如隐式记忆提取）

**技术栈清单：**
- Java 17+
- Spring Boot（核心应用框架）
- LangChain4j（LLM 编排与工具集成框架，用于与大模型交互）
- OpenAI / 兼容 OpenAI 格式的模型客户端
- Lombok（可选，简化实体和配置类）
- Jackson（JSON 读写，Spring Boot 默认集成）
- Spring Task / `@Scheduled`（定时任务）

### 2.3 存储文件结构说明

根据设计文档的六个核心区块，将存储分为六个文件（按用途划分为：元数据、对过去对话的自动摘要、用户档案卡（User Insights）、短期上下文、完整历史、队列状态）：

1. **metadata.json**：包含
   - User Interaction Metadata（客户端环境）
   - Assistant Response Preferences（回复风格偏好）
   - Global Controls（全局控制开关）

2. **model_set_context.json**：包含
   - Model Set Context（对过去几周或几个月的互动进行的**自动摘要**，以段落形式概括历史对话中最重要的内容，不存储原始对话文本）

3. **implicit_memories.json**：包含
   - User Insights（”用户档案卡”：主要存储用户的偏好、职业、背景、长期习惯等与用户相关的信息）
   - Notable Past Conversation Topics Highlights（可选：作为辅助结构，用于支撑或生成 User Insights 的条目）

4. **recent_user_messages.jsonl**：包含
   - Recent Conversation Content（最近40条用户消息，滚动窗口，仅包含用户消息）

5. **conversation_history.jsonl**：包含
   - 完整的对话历史日志（appendonly 模式，用于生成 Model Set Context 摘要）
   - 格式：每行使用管道分隔符（`timestamp|role|message`）

6. **memory_queues.json**：包含
   - young 队列和 mature 队列的持久化状态
   - 用于恢复 Top of Mind 算法的双向队列结构

## 3. 核心模块开发说明

### 3.1 模块一：系统提示词分层架构

#### 3.1.1 职责
将不同生命周期和类型的记忆数据结构化，并按类别注入到大模型的 System Prompt 中。

#### 3.1.2 六个核心区块

**1. User Interaction Metadata**
- **内容**：客户端环境（设备类型、明暗主题等）
- **数据来源**：用户输入或系统检测
- **存储位置**：`/.memory/metadata.json`

**2. Assistant Response Preferences**
- **内容**：系统级或用户全局设定的回复风格（如"以项目符号输出"、"保持简短"）
- **标签**：`Confidence: high`
- **存储位置**：`/.memory/metadata.json`（与 User Interaction Metadata 合并存储）

**3. Recent Conversation Content**
- **内容**：最近约 40 次用户输入的原始消息（不包括助手回复）
- **格式**：每行使用管道分隔符（`|`），格式为 `timestamp|message`（仅存储用户消息）
- **时间戳格式**：使用本地时间，格式为 `YYYY-MM-DD HH:MM:SS`
- **读取策略**：读取文件时，按行解析管道分隔的内容，得到完整的用户消息序列，由上层在构造对话时统一加上 `role: "user"`
- **存储位置**：`/.memory/recent_user_messages.jsonl`

**4. Model Set Context**
- **内容**：对过去几周或几个月的互动进行的**自动摘要**，通常是若干段落级别的自然语言总结
- **特点**：不存储原始对话文本，而是对历史对话进行概括，用于在 System Prompt 中快速恢复长期上下文
- **确认机制**：是否需要用户确认可根据实现决定（例如首次生成或大幅变更摘要时提示用户）
- **存储位置**：`/.memory/model_set_context.json`

**5. User Insights（用户档案卡）**
- **内容**：围绕“用户是谁”构建的档案信息，例如：
  - 职业、工作背景
  - 长期偏好（技术栈、写作风格、爱好等）
  - 常驻城市、时区等背景信息
- **结构**：**推荐一条用户信息对应一个独立的槽位 item**，例如：
  - `home_city`：用户当前常驻城市
  - `work_role`：用户当前的职业角色
  - `diet_preference`：饮食偏好
  这样可以方便后续新增、修改或覆盖单条档案信息，而不是把所有信息写进一个超长字符串。
- **角色**：作为用户画像的主要载体，便于在 System Prompt 中快速恢复与用户相关的长期信息
- **存储类型**：`memory_type: "user_insight"`
- **存储位置**：`/.memory/implicit_memories.json`
    
**6. Notable Past Conversation Topics Highlights（可选层）**
- **内容**：从长期对话中提炼出的、与用户强相关的主题与行为模式（例如“最近频繁讨论记忆管理产品体验”等）
- **角色**：可以作为“中间层”，用于支持或生成更高层次的 User Insights 条目；也可以在需要时直接注入 Prompt 以补充上下文
- **存储类型**：`memory_type: "notable_highlights"`
- **存储位置**：`/.memory/implicit_memories.json`

#### 3.3.2 Top of Mind 算法

**数据结构：双向队列（young / mature）**

- 参考 ChatGPT 的实现，采用**两层 LRU 双向队列（young / mature）**策略，类似 MySQL 的双层 LRU 淘汰：
  - **young 队列**：新写入的记忆、或长期未用后重新命中的记忆，首先进入 `young` 队列
  - **mature 队列**：只有在 `young` 队列中的记忆再次被访问时，才会被”升级”到 `mature` 队列，表示这是稳定的高价值记忆
  - 队列内部按照”最近最少使用（LRU）”顺序维护，队列头部为最近访问，尾部为最久未访问
- `hit_count` 和 `last_accessed` 仍然保存在 Memory 对象上，用于辅助维护 LRU 顺序与清理策略（例如 30 天未访问算长期）

**升级条件：**
- 当 `young` 队列中的记忆被再次访问时，将其移动到 `mature` 队列头部

**Top of Mind 选择流程：**

1. **列表内排序**：优先从 `mature` 队列按 LRU 顺序（从头部开始）选取前若干个槽位；如果不足，再从 `young` 队列按 LRU 顺序补足
2. **数量限制**：Top of Mind 记忆数量默认限制在 10–15 个之间（可配置）
