- 本文档是项目的**架构与实现规范**。
- 本项目只采用一种记忆模型：**四层架构**。
- 如与其他历史文档冲突，以本文档为准。

文档版本：v4.6  
最后更新：2026-03-31

## 2. 项目概述

### 2.1 项目目标

构建一个可持续演进的 CLI 记忆系统，让模型在多轮对话中实现：

1. 保持短期上下文连续性
2. 记住用户偏好与背景（可管理、可删除）
3. 在相似问题上通过语义检索复用历史经验

### 2.2 技术栈

- Java 17+
- Spring Boot 3.2.0
- LangChain4j 0.35.0
- OpenAI 格式 API
- all-MiniLM-L6-v2（本地嵌入）
- Jackson / Lombok

### 2.2.1 文件即记忆（Memory as Filesystem）定义

本项目不是“聊天+工具”的通用 Agent，而是“文件即记忆”的记忆系统：

1. 记忆载体统一为文件（`json/jsonl/md`）。
2. shell 操作是记忆操作原语：对文件记忆进行 CRUD、检索与治理（读/写/改/检索/归档）。
3. 系统核心贡献聚焦“面向文件记忆体的提取、组织、检索、治理、评测闭环”。

### 2.3 架构原则

1. 单一职责：类职责边界清晰，不做“万能类”。
2. 分层调用：`cli -> memory/rag -> llm -> prompt`，禁止反向依赖。
3. 统一写入：记忆落盘必须经过 `MemoryWriteService`。
4. 提取与通信解耦：`LlmClient` 只通信，`LlmExtractionService` 做提取业务。
5. 文件可恢复：存储文件可在启动时自动初始化与迁移。

### 2.4 毕设扩展原则

项目后续扩展以“保持 `Memory-System` 核心定位”为前提，遵循以下约束：

1. 不改动四层记忆架构基线，只在层内增强能力或补充横向治理模块。
2. 新功能优先回答“记忆如何被提取、组织、检索、验证、应用”，避免偏离成通用 Agent 平台。
3. 所有新增能力尽量落到现有模块边界：`memory / rag / task / im / prompt / tool`。
4. 论文贡献优先强调“系统性”和“可验证性”，因此评测、指标、审计链路是正式功能，不是附属脚本。
5. 文档、实现、实验计划必须同步更新，避免毕设阶段出现“代码有了但无法讲清楚”的问题。

## 3. 四层记忆架构

### 3.1 架构总览

```text
Layer 1: Short-term Memory
Layer 2: Metadata
Layer 3: User Insight
Layer 4: Learned Knowledge
         ├── 4a Skill (.md)
         └── 4b Example (Vector/RAG)
```

说明：

1. 四层架构是本项目唯一记忆基线。
2. 后续新增“摘要记忆、治理状态、评测、身份映射、主动服务”均视为在四层基线之上的增强能力，而不是新增第五层。

### 3.1.1 会话启动导航文件 `Agent.md`（非记忆层）

`Agent.md` 不属于四层记忆中的任何一层，它是会话启动阶段的导航文件。

职责限制：

1. 说明关键文件在哪里
2. 说明各文件的职责和加载方式
3. 标注哪些内容允许自动加载，哪些内容需要按需读取

约束：

1. 不存放大量知识正文
2. 不存放高频变动的运行态信息
3. 不存放完整用户记忆内容
4. 每次新会话启动前自动加载一次，不要求每轮重复全文注入

当前方案路径：

1. 仓库根目录 `Agent.md`：全局导航图
2. 后续如需本地覆盖，可再增加 `.memory/Agent.local.md`

### 3.2 Layer 1 - Short-term Memory（短期记忆）

#### 定义

当前会话的即时上下文，不作为长期画像沉淀。

#### 组成

1. 最近完整对话窗口（user + assistant）
2. 最近用户输入日志（user-only）

#### 当前实现

- 对话上下文：`conversation_history.jsonl`
- 用户输入窗口：`recent_user_messages.jsonl`
- 运行时策略：
  - messages：默认最近 10 轮完整对话
  - prompt：默认注入第 11~40 轮用户消息（连续性日志）

#### 相关代码

- `MemoryStorage.getRecentConversationTurns()`
- `MemoryStorage.getOlderUserMessages()`
- `ConversationCli.processUserMessage()`

### 3.3 Layer 2 - Metadata（元数据）

#### 定义

描述用户环境、助手偏好、全局控制开关的静态或半静态信息。

#### 存储

- 文件：`metadata.json`
- 典型字段：
  - `user_interaction_metadata`
  - `assistant_preferences`
  - `global_controls`（`use_saved_memories`、`use_chat_history`）
  - `ui_settings`（`cli_theme`、`show_footer`）

#### 当前实现

- 读取：构建 system prompt 前读取
- 更新：CLI 命令 `/controls` 修改 `global_controls`
- 更新：CLI 命令 `/theme`、`/footer` 修改并持久化 `ui_settings`

#### 相关代码

- `MemoryStorage.readMetadata()/writeMetadata()`
- `CliRunner.showControls()/setControl()`
- `ConversationCli.buildSystemPrompt()`

---

### 3.4 Layer 3 - User Insight（用户洞察）

#### 定义

用于提升后续回复质量的用户偏好、背景、长期习惯信息。

#### 数据模型

当前统一使用 `Memory` 对象，关键字段：

- `content`
- `memoryType`（当前仅 `USER_INSIGHT`）
- `source`（`EXPLICIT` / `IMPLICIT`）
- `confidence`
- `hitCount`
- `createdAt` / `lastAccessed`

#### 存储

1. `.memory/user-insights.md`：用户洞察单文档正文
2. `pending_explicit_memories.jsonl`：显式提取冲突队列
3. `memory_queues.json`：当前实现兼容保留，后续视单文档方案决定是否退役
4. `user_insights.json`：迁移源，完成迁移后废弃

#### 写入来源

1. 显式提取：用户消息实时检测
2. 隐式提取：夜间任务或 `/memory-update`

#### 当前实现

- 已实现显式提取 + 冲突入队
- 已实现隐式提取（用户洞察）
- 已实现 Top of Mind 访问升级与淘汰
- 已实现 `.memory/user-insights.md` 单文档画像模型
- 启动时支持 `user_insights.json -> user-insights.md` 自动迁移

#### 目标文件格式

用户洞察正文文件建议使用单个 Markdown 文档，保存稳定内容和可读描述。

```md
---
updated_at: 2026-03-18T10:00:00+08:00
style: narrative
---

用户当前的长期画像如下。

用户不爱吃鱼，在推荐菜谱、餐厅和聚餐方案时应避开鱼类。用户更偏好简洁直接的表达，不喜欢冗长说明。用户当前正在开发一个记忆系统项目，讨论重点集中在本地文件系统记忆、工具加载和分层架构收敛。
```

说明：

1. `User Insight` 作为单文档维护，不再按槽位拆分多个 Markdown 文件
2. 正文使用自然段，不使用项目符号列点表达用户画像
3. 文档面向人工阅读和整体重写，不再以单槽位增删改为主
4. 当前 `hitCount`、`lastAccessed`、队列状态仍由旧实现维护；迁移后需重新评估是否保留

#### 相关代码

- `MemoryExtractor`
- `LlmExtractionService.extractExplicitMemory()/extractUserInsights()`
- `MemoryWriteService`
- `MemoryManager`

### 3.5 Layer 4 - Learned Knowledge（持续学习知识）

#### 定义

沉淀“怎么做事”的知识，不是“用户是谁”。

#### 4a Skill（方法论）

- 形态：`.memory/skills/*.md`
- 作用：沉淀稳定策略、最佳实践
- 加载方式：通过固定工具 `load_skill(name)` 按需读取
- 当前状态：**已接入对话主链路，默认不再全量注入 skill 正文**

#### 4b Example（案例）

- 形态：`vector_store.json`
- 作用：通过向量语义检索召回相似历史片段
- 当前状态：**已实现基础能力**

当前实现能力：

1. 记忆向量化索引（`indexMemory`）
2. 批量重建索引（`indexAllMemories`）
3. 语义搜索（`searchMemories`）
4. 对话中构建 RAG 上下文（`buildSmartContext`）

## 4. 会话启动与 Prompt 组装策略

### 4.1 会话启动阶段

新会话启动前先读取 `Agent.md`，建立最小导航上下文。

启动阶段规则：

1. 自动加载 `Agent.md`
2. 在启用记忆时加载 `user-insights.md`
3. 不自动全文加载所有 `Skill` 文件
4. 需要正文时再按需读取目标文件

### 4.2 对话轮次 Prompt 组装

`SystemPromptBuilder` 当前组装顺序：

1. `Agent.md` 导航摘要或启动缓存
2. Layer 2 Metadata
3. Layer 2 Assistant Preferences
4. Layer 1 用户输入日志（older user messages）
5. Layer 3 User Insights（单文档画像正文）
6. Layer 4a Skills（按需加载）
7. Layer 4b RAG Context / Examples

同时，Layer 1 最近完整对话通过 `messages` 单独传递给模型。

说明：

1. Prompt 负责把多层记忆组织成可被模型消费的上下文。
2. 后续若新增摘要记忆、检索解释、记忆状态标签，也应优先作为 Prompt 组装策略增强，而不是破坏现有主流程。

## 5. 面向毕设的扩展模块设计

### 5.1 下一阶段总目标

下一阶段不采用显式“记忆路由器”方案，而是建设更高级的 `Memory Reflection` 机制。

核心目标：

1. 让 Agent 在回答前先判断“当前问题是否真的需要过去的信息”。
2. 让 Agent 明白“为什么需要记忆”，而不是因为系统里有记忆就机械调用。
3. 让每次记忆调用都带有证据目的，例如个性化、连续性、约束遵循、经验复用、行动跟进。
4. 让系统对外可展示“本轮回答依赖了哪些过去依据”，增强可解释性和答辩表现力。

系统定位保持不变：

1. 项目仍然是四层 `Memory-System`。
2. `Memory Reflection` 属于回答前的推理增强机制，不新增记忆层。
3. 它负责决定“记忆是否必要”和“记忆调用目的”，不是传统按模块分流的 Router。

### 5.2 下一阶段核心能力一：Memory Reflection（记忆反思机制）

目标：让系统在生成回答前，先进行一次面向回答目标的自我反思，判断是否需要记忆支撑。

#### 5.2.1 设计原则

1. 不是先问“查哪个模块”，而是先问“只靠当前输入是否能回答好”。
2. 如果不能回答好，再判断缺失的是哪一类“过去证据”。
3. 记忆调用应服务于回答质量提升，而不是固定流程中的强制步骤。
4. 反思结果必须可落盘、可调试、可展示。

#### 5.2.2 反思输出结构

建议新增结构化结果对象 `MemoryReflectionDecision`，至少包含：

- `needsMemory`：是否需要调用长期记忆
- `memoryPurpose`：记忆使用目的
- `reason`：自然语言解释
- `confidence`：判断置信度
- `retrievalHint`：对后续检索的提示语
- `evidenceTypes`：预期需要的证据类型

其中 `memoryPurpose` 建议枚举：

- `PERSONALIZATION`
- `CONTINUITY`
- `CONSTRAINT`
- `EXPERIENCE_REUSE`
- `ACTION_FOLLOWUP`
- `NOT_NEEDED`

其中 `evidenceTypes` 建议枚举：

- `USER_INSIGHT`
- `SESSION_SUMMARY`
- `TASK`
- `EXAMPLE`
- `SKILL`
- `RECENT_HISTORY`

#### 5.2.3 触发时机

建议流程插入到 `ConversationCli.processUserMessage()` 中，在组装 system prompt 与执行 RAG/Skill 之前：

1. 接收用户输入
2. 运行 `Memory Reflection`
3. 若 `needsMemory=false`，则尽量基于当前轮和最近上下文回答
4. 若 `needsMemory=true`，则按 `retrievalHint` 与 `evidenceTypes` 组织记忆上下文
5. 生成回复
6. 记录本轮使用了哪些记忆证据

#### 5.2.4 需要覆盖的问题类型

下一阶段至少覆盖以下六类问题：

1. 用户身份与偏好类
   - 例如“按我的习惯推荐”“我之前喜欢什么”
2. 跨会话延续类
   - 例如“继续上次的毕设规划”
3. 任务与提醒类
   - 例如“我还有哪些没做”“提醒我明天继续”
4. 历史经验复用类
   - 例如“之前类似问题是怎么解决的”
5. 方法与策略类
   - 例如“这类问题一般怎么系统处理”
6. 一般知识问答类
   - 例如普通技术概念解释，通常不需要长期记忆

#### 5.2.5 失败保护

1. 即使 `Memory Reflection` 输出异常，主对话链路仍必须可继续。
2. 当结构化结果解析失败时，回退为当前默认行为。
3. 不允许因为反思阶段失败导致整个问答不可用。

### 5.3 下一阶段核心能力二：记忆证据视图（Memory Evidence View）

目标：系统不仅回答，还能说明“这次回答为何参考了过去”。

#### 5.3.1 设计要求

每一轮回答完成后，系统内部至少应能记录：

1. 本轮是否触发了长期记忆调用
2. 触发目的是什么
3. 调用了哪些证据类型
4. 实际命中了哪些记忆内容
5. 哪些记忆最终真正参与了回答

#### 5.3.2 输出结构

建议新增 `MemoryEvidenceTrace`，包含：

- `userMessage`
- `reflectionDecision`
- `retrievedInsights`
- `usedInsights`
- `retrievedExamples`
- `usedExamples`
- `loadedSkills`
- `usedSkills`
- `retrievedTasks`
- `usedTasks`
- `usedEvidenceSummary`
- `timestamp`

#### 5.3.3 展示要求

CLI 至少提供一个调试入口，例如：

- `/memory-debug`

输出内容建议包括：

1. 本轮是否需要记忆
2. 为什么需要或为什么不需要
3. 记忆使用目的
4. 命中的记忆条目与分数
5. 最终引用的证据摘要
6. 按证据类型展示 `retrieved` 与 `used` 的 Top-N 条目，便于解释“检索到”与“真正使用”的差异

说明：

1. 该视图首先服务开发和答辩展示，不要求默认对普通用户可见。
2. 后续如接入 IM，可考虑仅在 debug 模式下返回。

### 5.4 下一阶段核心能力三：会话摘要与情节记忆

目标：让系统不仅记住“事实”，也记住“过程与脉络”。

#### 5.4.1 需求描述

1. 当会话长度达到阈值时，自动生成 `session summary`。
2. 当话题发生明显迁移时，生成 `topic summary`。
3. 当项目阶段、任务状态、用户目标发生重大变化时，生成 `milestone summary`。
4. 这些摘要优先作为连续性证据参与 `Memory Reflection` 之后的回答构建。

#### 5.4.2 存储建议

- `.memory/session_summaries.jsonl`
- `.memory/topic_summaries.jsonl`
- `.memory/milestone_summaries.jsonl`

每条摘要至少包含：

- `summaryId`
- `summaryType`
- `content`
- `sourceRange`
- `createdAt`
- `keywords`
- `canonicalUserId`（若后续接入多用户）

#### 5.4.3 使用规则

1. 当用户提问明显依赖历史延续时，优先注入摘要，而不是全文长历史。
2. 摘要不足时，再补充最近对话窗口与相关原始记录。
3. 摘要应被视为高价值连续性证据，而不是普通日志缓存。

### 5.5 下一阶段核心能力四：案例蒸馏链路

目标：让系统具备“从具体经验抽象出方法”的成长能力。

#### 5.5.1 基本链路

下一阶段建议引入三层成长对象：

1. `Example`
   - 原始问题与解决方案
2. `Pattern`
   - 多个 Example 的共性步骤、适用边界、常见失败点
3. `Skill`
   - 已验证、可复用的正式方法文档

#### 5.5.2 需求描述

1. 成功解决的问题可沉淀为 `Example`。
2. 当多个 `Example` 高度相似时，系统可尝试生成 `Pattern`。
3. 当 `Pattern` 稳定并被多次复用时，可进一步蒸馏成 `Skill`。
4. `Skill` 仍保持按需加载，不重新变回全量注入模式。

#### 5.5.3 创新点表述

这一能力的重点不是“存更多数据”，而是：

1. 让 Memory-System 从记住案例进化到总结方法。
2. 体现系统具备长期学习和知识升维能力。
3. 为毕设提供区别于普通 RAG 系统的明显创新点。

### 5.6 下一阶段核心能力五：场景化展示能力

目标：展示重点不是“有无记忆对比”，而是“同一系统在不同使用场景下如何发挥作用”。

#### 5.6.1 必须覆盖的展示场景

下一阶段至少准备以下五类演示场景：

1. 个性化建议场景
   - 展示系统如何根据长期偏好生成定制回答
2. 跨会话连续协作场景
   - 展示系统如何承接上一次项目讨论和任务状态
3. 多端统一记忆场景
   - 展示 CLI、飞书、Telegram 之间的记忆连续性
4. 主动服务场景
   - 展示系统如何基于历史任务和目标主动提醒或建议
5. 案例蒸馏场景
   - 展示系统如何从多个 Example 形成 Pattern 或 Skill

其中主动服务场景至少覆盖两类固定展示：

1. 定时提醒型
   - 例如用户说“下周提醒我提交毕设初稿”，到时间后系统主动提醒
2. 记忆驱动建议型
   - 例如系统根据近期主题和任务进展，主动生成“你最近一直在推进记忆治理模块，建议先完成数据结构设计”

#### 5.6.2 展示形态要求

即使不做 Web UI，CLI 也应具备结构化展示能力。

建议新增命令：

1. `/memory-debug`
   - 展示反思决策与证据调用链（支持 `/memory-debug [N]` 查看最近 N 轮）
   - 历史 trace 中 `needs_memory` 采用三态展示：`是/否/unknown`（缺字段不默认映射为“否”）
2. `/memory-timeline`
   - 展示偏好、任务、主题的时间演化
3. `/memory-review`
   - 展示待确认、冲突、低置信度记忆
4. `/memory-report`
   - 输出当前用户的记忆总览与统计信息
5. `/memory-scenes`
   - 输出当前可用于答辩的场景脚本和数据状态
6. `/tasks`
   - 输出待办任务、即将到期任务、最近触发任务和主动建议来源
7. `/memory-insights [limit]`
   - 输出证据检索/使用率、反思命中率和可执行优化建议

#### 5.6.3 答辩展示要求

展示时不应只停留在聊天结果，而应明确表现：

1. 用户当前处于什么场景
2. 系统为什么需要过去的信息
3. 调用了哪些过去证据
4. 最终回答如何受到这些证据影响
5. 系统有没有把新的经验再次沉淀回记忆

对于主动服务类展示，还必须明确表现：

1. 任务是如何从自然语言中被识别和创建的
2. 到时间后系统如何主动触发提醒
3. 系统主动建议依赖了哪些长期记忆、摘要或任务状态
4. 主动输出不是随机生成，而是由记忆证据驱动

### 5.7 下一阶段核心能力六：自然语言定时任务与主动服务

目标：让系统从“能保存任务”升级为“能理解自然语言中的任务需求，并在合适时机主动服务”。

#### 5.7.1 功能范围

下一阶段需要同时支持两种能力：

1. 用户通过自然语言创建定时任务
2. 系统基于记忆与任务状态主动生成提醒或建议

#### 5.7.2 自然语言定时任务

需求描述：

1. 当用户表达时间相关意图时，系统应能识别任务创建需求。
2. 允许用户使用自然语言直接下达任务，例如：
   - “下周提醒我提交毕设初稿”
   - “明天下午三点提醒我给导师发消息”
   - “周五晚上提醒我整理答辩 PPT”
3. 系统需要通过 LLM 将自然语言解析为结构化定时任务。
4. 若信息不完整，可由模型给出合理默认值，或要求补充最少必要信息。
5. 已创建任务必须可查询、可去重、可触发、可追溯来源。

建议字段：

- `title`
- `detail`
- `dueAt`
- `sourceMessage`
- `sourcePlatform`
- `sourceConversationId`
- `sourceSenderId`
- `taskIntent`
- `taskCategory`

说明：

1. 该能力已具备基础 `create_task(...)` 工具链路，下一阶段重点是把它正式纳入毕设主流程和展示脚本。
2. 自然语言建任务必须被视为 Memory-System 的组成能力，而不是孤立工具能力。

#### 5.7.3 主动提醒与主动建议

需求描述：

1. 当任务到期时，系统应主动向原会话或原平台发出提醒。
2. 除到期提醒外，系统还应支持基于长期记忆和近期摘要生成“主动建议”。
3. 主动建议必须依赖明确的记忆证据，例如：
   - 最近持续出现的项目主题
   - 长期未完成的任务
   - 即将到来的里程碑
   - 用户一贯偏好的工作方式
4. 主动建议应尽量具体、可执行，避免空泛的鸡汤式输出。

主动建议示例：

- “你最近一直在推进记忆治理模块，建议先完成数据结构设计，再补 CLI 审核命令。”
- “你本周已经多次讨论答辩材料，建议今天先整理场景化展示脚本。”

#### 5.7.4 主动服务的展示要求

主动服务场景在答辩中必须能展示三件事：

1. 任务如何由自然语言生成
2. 系统如何在正确时间主动提醒
3. 系统如何基于记忆主动给出阶段性建议

#### 5.7.5 与现有模块的关系

建议落点：

1. `task/ScheduledTaskService.java`
2. `tool/CreateTaskTool.java`
3. `llm/LlmExtractionService.java`
4. `cli/ConversationCli.java`
5. `im/ImRuntimeService.java`

说明：

1. 任务解析属于 LLM 结构化提取能力。
2. 任务触发属于调度与消息推送能力。
3. 主动建议属于 Memory Reflection 之后的主动服务扩展能力。

#### 5.7.6 飞书主动提醒增强调研（2026-03-29）

调研目标：将 Task 从“到点提醒一次”升级为“可追踪、可互动、可闭环”的飞书主动提醒能力。

当前实现现状（已具备）：

1. 已具备飞书文本消息发送能力（`im.message.create` 文本）。
2. 已具备飞书入站能力（长连接 + webhook 事件解析）。
3. 已具备任务到期触发与推送主链路（`ScheduledTaskReminderJob`）。
4. 已具备跨平台统一身份与会话绑定能力，可用于推送目标定位。

调研结论（基于飞书开放平台能力与当前工程可行性）：

1. 飞书主动提醒可分为三层能力：
   - `消息发送层`：文本/富文本/交互卡片
   - `交互回传层`：用户在卡片上点击“完成/延期”等动作
   - `状态更新层`：回写任务状态并更新后续提醒策略
2. 当前工程距离“完整主动提醒”主要缺口不在接入，而在：
   - 任务状态机不够细（仅 pending/triggered/cancelled）
   - 缺少“提前提醒/重复提醒/升级提醒”策略
   - 缺少飞书交互卡片回调处理（按钮动作无法闭环）

建议能力矩阵（优先级从高到低）：

1. P1（高优先级，1 周内可落地）— 多阶段主动提醒
   - 增加提醒时机：`T-24h`、`T-1h`、`DueNow`、`OverdueDaily`
   - 增加任务状态：`PENDING / IN_PROGRESS / DONE / SNOOZED / OVERDUE / CANCELLED`
   - 在 `/tasks` 中展示“下次提醒时间、已提醒次数、最近提醒渠道”

2. P2（高优先级，1~2 周）— 飞书交互卡片提醒
   - 任务提醒改为卡片（按钮：完成、延期 1 小时、明天提醒、忽略）
   - 卡片中展示：任务标题、截止时间、来源会话、推荐动作
   - 卡片发送失败自动降级为文本提醒（保证稳定性）

3. P3（中优先级，2 周）— 交互回传闭环
   - 新增卡片回调入口（接收按钮动作）
   - 回调后更新任务状态并写入操作审计
   - 同步更新周报统计（完成率、拖延率、平均响应时长）

推荐数据结构扩展（Task 维度）：

- `status`：扩展为 `PENDING/IN_PROGRESS/DONE/SNOOZED/OVERDUE/CANCELLED`
- `nextReminderAt`
- `lastRemindedAt`
- `reminderCount`
- `reminderPolicy`（如 `at_due`、`before_1h`、`before_24h`、`overdue_daily`）
- `feishuMessageId`（用于卡片更新）
- `lastActionBy` / `lastActionAt` / `lastActionType`（交互审计）

推荐模块落点：

1. `task/ScheduledTaskService.java`
   - 扩展状态机、提醒策略计算、延期逻辑
2. `task/ScheduledTaskReminderJob.java`
   - 从“到点触发”改为“策略触发”（支持提前/逾期提醒）
3. `im/feishu/FeishuOutboundClient.java`
   - 新增 `sendCard(...)`，保留 `sendText(...)` 兜底
4. `im/feishu/FeishuWebhookController.java`（或新增 callback controller）
   - 处理卡片按钮回调，驱动任务状态变更
5. `cli/CliRunner.java`
   - 增强 `/tasks` 输出（状态、提醒计划、交互轨迹）

验收标准（建议）：

1. 用户在飞书收到任务卡片提醒，可点击“完成/延期”。
2. 点击后任务状态在系统中实时更新，并可在 `/tasks` 查询。
3. 对逾期任务可按策略自动再次提醒，且不产生重复轰炸。
4. 周报中可看到“完成任务数、拖延任务数、提醒响应率”。

1. [飞书开放平台文档首页](https://open.feishu.cn/document/)
2. [IM 消息发送（im.message.create）](https://open.feishu.cn/document/server-docs/im-v1/message/create)
3. [事件订阅指南](https://open.feishu.cn/document/server-docs/event-subscription-guide/overview)

### 5.8 下一阶段核心能力七：记忆治理与可信度增强

目标：保证被系统调用的记忆足够可信、可审计、可修正。

建议扩展字段：

- `status`：`ACTIVE / PENDING / ARCHIVED / FORGOTTEN`
- `sourcePlatform`
- `sourceMessageId`
- `lastVerifiedAt`
- `conflictWith`
- `verificationNote`
- `confidenceDecayScore`

需求描述：

1. 冲突记忆进入审核队列，而不是直接覆盖。
2. 低置信度记忆应允许进入待验证状态。
3. 同一记忆被多次证实时，可提升稳定度。
4. 长期未被证实的记忆可逐步衰减。

说明：

1. 该模块为 `Memory Reflection` 提供可信记忆基础。
2. 没有治理能力的记忆系统，后续反思与主动服务都容易失真。

建议评测维度：

1. `Memory Reflection` 判断准确率

4. 案例蒸馏质量
5. 主动提醒有效性
6. 记忆证据解释完整性

建议产物：

1. `scripts/` 下场景化回放脚本
2. `src/test/` 下固定问答样例与期望输出
3. CLI 证据洞察命令（建议：`/memory-insights [limit]`）
4. 可直接用于论文的统计口径和实验说明

#### 5.9.1 实用增强：`/memory-insights`（证据质量洞察）

设计目标：

1. 把 `memory_evidence_traces.jsonl` 从“日志”提升为“可运营数据”
2. 直接回答“检索到了什么、真正用了什么、为什么没用上”
3. 为 prompt 调优与检索策略调整提供可执行建议

设计思路（搜索与收敛）：

1. 先复用已有 `MemoryTraceInsightService` 聚合能力，避免重复实现统计逻辑
2. 在 `CliRunner` 暴露 `/memory-insights [limit]`，提供近 N 条窗口分析
3. 输出统一四类证据的 `used/retrieved` 使用率：`insights/examples/skills/tasks`
4. 增加用途诊断（按 `evidence_purpose` 分组），定位“哪类场景记忆效果差”
5. 增加窗口趋势对比（前半窗口 vs 后半窗口），直接识别“最近效果变好/变差”
6. 同时输出高频 `skill/purpose/reason` 和建议列表，形成“指标 -> 诊断 -> 改进”闭环

输出建议（CLI）：

1. 样本窗口：`requestedLimit` 与 `sampleSize`
2. 反思命中：`needs_memory` 比例、`memory_loaded` 比例、`unknown` 比例
3. 证据使用率：四类证据的 `used/retrieved` 与百分比
4. 用途诊断：按 `purpose` 输出样本数、加载率、四类证据使用率
5. 趋势对比：前半窗口 vs 后半窗口（`memory_loaded` 与四类证据使用率）
6. 高频模式：Top `skill/purpose/reason`
7. 优化建议：按当前使用率阈值与数据质量信号输出下一步动作

### 5.10 下一阶段核心能力九：内容化产出与知识运营（Step 6/6 调研结论）

目标：将“记忆系统运行数据”持续转化为“可复现内容资产”，支持开发迭代、论文写作、答辩演示三条线并行推进。

调研结论（围绕当前代码与数据资产）：

1. 内容原材料已齐备，短板在生产机制而非数据来源。
2. 内容生产应坚持“命令驱动 + 模板固化 + 版本留痕”，避免人工自由发挥导致口径漂移。
3. 最低成本路径是先做 CLI 侧模板化与目录规范，再决定是否补充导出脚本或 UI。

内容支柱（建议长期固定）：

4. 研究支柱（Research）

建议新增内容资产池（优先级）：

1. 周复盘卡（P1）
2. 场景案例卡（P1）
3. 任务闭环卡（P1）

模板统一字段（所有卡片最少包含）：

1. 时间窗口（`from/to`）
2. 数据来源（命令与文件路径）
3. 关键指标（最多 5 项）
4. 证据片段（trace id / 轮次 / 场景名）

建议目录规范（新增）：

1. `.memory/content-assets/weekly/`：周复盘卡
2. `.memory/content-assets/scenes/`：场景案例卡
3. `.memory/content-assets/tasks/`：任务闭环卡

#### 5.10.1 内容选题矩阵（围绕记忆系统主线）

建议将高频内容选题固定为以下 10 类，避免“每周重新想题”导致效率损失：

1. 反思准确性（`needs_memory` 判定正确率、误判样例）
2. 证据利用率（`retrieved -> used` 转化差距）
3. 个性化质量（同问题“有画像/无画像”输出差异）
4. 连续性质量（摘要命中后跨会话承接效果）
5. 治理健康度（冲突、待审核、低置信度占比变化）
6. 任务闭环效率（创建到完成链路时长、延期率）
7. 主动服务有效性（提醒触达率、建议采纳率）
8. 场景失败复盘（失败触发条件、修复动作、复测结果）
9. 结构化兼容性（legacy trace 解析成功率）
10. 论文口径沉淀（图表口径与 CLI 指标一致性）

#### 5.10.2 命令驱动内容流水线（SOP）

建议固化“采集 -> 草稿 -> 复核 -> 入库”的最小流程：

   - 执行 `/memory-insights [limit]`、`/memory-review`、`/memory-scenes`、`/tasks`

4. 入库（1 分钟）

2. 中段必须给出“证据链路”（命令、trace、关键字段）
3. 结尾必须给出“改进动作 + 下一次验证时间”
4. 禁止无数据结论；每条结论至少绑定一个可回放证据

内容效果 KPI（最低门槛）：

1. 产出 KPI：每月 >= 8 条可发布内容（含 >= 2 条长内容）
2. 质量 KPI：每条内容至少 1 个可复现实验或回放脚本
3. 闭环 KPI：每月 >= 3 条内容反向进入开发 backlog 并完成验证
4. 复用 KPI：答辩与论文材料中 >= 70% 图表可直接复用内容资产

#### 5.10.5 内容增长飞轮（调研深化）

目标：让“内容产出”不只服务记录，还能持续反哺系统优化与对外影响力。

飞轮结构（固定链路）：

1. 采样：从 `trace/insights/tasks/reminders` 抽取本周高信号样本
2. 归因：定位“为什么成功/失败”（reflection、evidence、治理状态）
3. 产品化：拆分为 L1/L2/L3 内容并绑定受众
4. 反馈：收集评论/私信/复现实验反馈，沉淀可执行问题
5. 回流：转成 backlog，进入修复和复测，再产生下一批样本

内容来源映射（建议固定）：

1. `/memory-insights` -> 趋势和回归 -> 周复盘卡 / 指标解读短内容
2. `/memory-debug [N]` -> 单轮证据链 -> 失败复盘卡 / 场景拆解文
3. `/memory-scenes` -> 跨轮场景承接 -> 连续性案例卡 / 演示脚本
4. `/tasks` + 主动提醒日志 -> 闭环效率 -> 任务闭环卡 / 主动服务案例
5. `src/test/**` + 回放脚本 -> 可复现实验 -> 论文图表卡 / 方法说明文

#### 5.10.6 内容实验框架（确保“可增长”）

建议围绕内容本身做轻量实验，避免只追求产量：

1. 标题实验：同一结论采用“问题导向/结果导向”两种标题，比较点击与互动
2. 结构实验：同一案例比较“先结论后证据”与“先问题后修复”留存差异
3. 证据密度实验：比较“1 个证据点”与“3 个证据点”对复现率的影响
4. 受众实验：同一素材分别面向开发者/答辩老师两类叙述，比较理解成本

实验记录最小字段（建议）：

1. `experiment_id`
2. `hypothesis`
3. `variant_a / variant_b`
4. `success_metric`（点击率、收藏率、复现成功率、反馈转 backlog 数）
5. `result`（保留/淘汰/继续观察）

#### 5.10.7 内容资产治理补充（避免失控）

建议在内容资产层补三条治理约束：

1. 新鲜度约束：超过 30 天未复测的结论标记 `stale`，发布前必须复测
2. 去重约束：同主题 14 天内不得重复发布“同证据同结论”内容
3. 撤回约束：发现结论不可复现时，72 小时内补“更正卡”或撤回原内容

建议在 `.memory/content-assets/index.md` 增加字段：

1. `content_id`
2. `pillar`（quality/scene/task/research）
3. `publish_level`（L1/L2/L3）
4. `evidence_refs`
5. `verification_status`（verified/stale/retracted）
6. `backlog_refs`

9. 历史 trace 回读需兼容 legacy 扁平字段：当 `needs_memory/reason/...` 位于顶层（无 `reflection` 对象）时，应保持可解析并沿用三态语义。
10. 历史 trace 回读需兼容“字符串化 JSON”格式：当 `reflection` 或 `retrieved/used` 列表被存为 JSON 字符串（如 `"[]"`、`"{...}"`）时，`/memory-debug` 仍应正确解析并展示，不得退化为原始字符串。
11. 反思服务异常返回 `null` 时，trace 侧仍需写入稳定反思结果：`use_saved_memories=true` 回退为 `ReflectionResult.fallback()`，`use_saved_memories=false` 固定为 `ReflectionResult.memoryDisabled()`，禁止出现空指针或语义漂移。
12. `/memory-debug` 与 `/memory-debug [N]` 需展示反思关键字段（`memory_purpose/confidence/retrieval_hint`），并在覆盖率较低时给出诊断提示（如“已检索但未使用”“使用偏低”），便于快速定位证据浪费。
13. 历史 trace 回读时 `reflection.confidence` 需兼容两类异常量纲：`1.0~2.0` 视为轻微越界并钳制为 `1.0`，`2.0~100` 视为百分制并归一化到 `0~1`，避免 `/memory-debug` 展示异常低置信度。

4. 对高价值、高频资源使用固定工具；对低频杂项资源使用 shell

## 7. 核心流程

### 7.1 对话主流程

1. 读取全局控制（是否启用记忆/历史）
2. 构建 system prompt（Agent 导航 + Metadata + 历史日志 + User Insight + 按需 Skill + RAG）
3. 读取最近完整对话作为 messages
4. 调用 LLM；如需要 skill 正文，通过 `load_skill(name)` 工具按需补充
5. 异步写入 recent/history
6. 异步执行显式提取与访问更新

### 7.2 显式记忆流程

1. 从当前用户消息提取候选记忆
2. 如槽位冲突，写入 `pending_explicit_memories.jsonl`
3. 如无冲突，更新 `user-insights.md` 的画像正文
4. 当前实现仍兼容更新 Top of Mind 和 RAG 索引；迁移后需重新评估

### 7.3 隐式提取流程

触发方式：

1. 夜间定时任务（默认 02:00）
2. CLI 手动命令 `/memory-update`

执行步骤：

1. 抽取近期历史
2. 提取用户洞察列表
3. 合并并重写 `user-insights.md` 正文

### 7.4 RAG 检索流程

1. 用户输入向量化
2. 检索向量库
3. 过滤已注入的上下文内容
4. 将相关内容注入 prompt

1. 非临时模式下真实模型/API 的端到端工具调用回归：已提供手动 E2E 用例与脚本，尚未沉淀稳定基线结果
2. `model_set_context.json` 尚未接入 runtime 主链路
3. `memory_queues.json` 在单文档画像方案下是否退役仍未定案
4. `MemoryType` 目前仅 `USER_INSIGHT`，后续可扩展
5. 缺少系统化自动测试用例

### 10.1 毕设展示优先新增功能（建议优先级）

1. 任务闭环看板（高优先级）
   - 新增 `/tasks today|overdue|done` 视图与完成/延期操作
   - 形成“创建 -> 提醒 -> 完成 -> 周复盘”的完整闭环

2. 记忆冲突处理命令（高优先级）
   - 新增 `/memory-approve <slot>`、`/memory-reject <slot>`
   - 将现有治理展示从“可查看”升级为“可操作”

3. 飞书交互卡片化（中高优先级）
   - 周报与任务提醒升级为飞书卡片
   - 提供完成/延期/忽略按钮，增强产品化展示效果

4. 团队空间权限与共享策略（中优先级）
   - 在 `team:*` 作用域增加只读/可写成员策略
   - 增加团队共享任务池与个人任务隔离规则

5. 一键答辩演示模式（中优先级）
   - 新增 `/demo run`，串联“无记忆 -> 个人记忆 -> 团队记忆 -> 周报推送”
   - 提升现场演示稳定性和可复现性

### 10.2 真实 API E2E 执行方式

1. 设置 `OPENAI_API_KEY`（可选设置 `OPENAI_BASE_URL`、`OPENAI_MODEL`）
2. 执行 `./scripts/run-real-api-e2e.sh`
3. 查看 `logs/e2e/real-api-e2e.YYYY-MM-DD_HH-MM-SS.log` 中的回归结果
