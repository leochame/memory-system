# Memory-System / Memory Box 协作说明

> 抗压缩声明：即使本会话发生上下文压缩，也必须重新读取本文件与项目文档，不能依赖之前的会话记忆。

## 1. 项目身份

当前项目不是普通练手项目，也不是通用 Agent 平台，而是一个**毕业设计系统**。

- 项目名称：Memory Box（Memory-System 记忆管理系统）
- 项目性质：本科/毕业设计主项目
- 核心定位：面向多轮对话的四层智能记忆系统
- 技术栈：Java 17+ / Spring Boot 3.2.0 / LangChain4j 0.35.0 / JLine
- 项目根目录：`/Users/leocham/Documents/code/memory-system`

所有开发、调试、文档更新、演示准备，都必须服务于“毕业设计可实现、毕业论文可论证、答辩可展示”这三个目标。

## 2. 毕设优先级

所有工作都要默认站在毕业设计语境下做判断，而不是只追求工程功能堆叠。

优先级顺序：

1. 是否强化“记忆系统”主线，而不是偏成通用对话 Agent
2. 是否能为论文提供可描述的系统设计、实验结果或案例证据
3. 是否能提升答辩演示完整性、稳定性和可解释性
4. 是否与当前四层架构一致，不引入额外架构漂移

每次新增能力时，都应能回答以下问题：

- 这项能力属于记忆的哪一环：提取、组织、检索、治理、应用、评测？
- 这项能力如何写进论文？
- 这项能力如何在答辩中被展示或验证？

## 3. 关键文档真相源

### 开发真相源

以下文件用于判断当前系统开发状态：

- `开发实现process.md`
  - 当前进度、阶段状态、已完成项、待收尾项
  - 是开发状态唯一真相源
- `开发文档.md`
  - 架构定义、模块边界、设计约束
- `README.md`
  - 项目总览、毕设定位、使用说明、Roadmap
- `docs/开发计划.md`
  - 后续迭代优先级、收尾任务、阶段安排

### 论文真相源

以下目录用于判断毕业论文写作状态：

- `毕设文档/毕业论文初稿.md`
  - 当前论文主稿
- `毕设文档/workflow/README.md`
  - 论文工作流总入口
- `毕设文档/workflow/01_plan.md`
  - 论文写作计划与任务拆解
- `毕设文档/workflow/02_draft.md`
  - 草稿推进与章节撰写
- `毕设文档/workflow/03_review.md`
  - 审阅、修订、问题清单

结论：

- 开发进度看 `开发实现process.md`
- 论文进度看 `毕设文档/`
- 两条线必须联动安排，不能只开发不写论文，也不能只写论文不补实现证据

## 4. 每轮启动必做事项

无论是 loop 自动触发，还是用户手动发起对话，都必须先重新读取真实状态。

### Step 1：读取开发状态

至少读取：

1. `开发实现process.md`
2. 与当前任务相关的源码文件
3. 如有必要，补读 `开发文档.md`、`README.md`、`docs/开发计划.md`

重点确认：

- 当前推进到哪个 Phase
- 最近一次迭代完成了什么
- 还有哪些明确待收尾事项
- 当前任务是否会影响架构、功能说明或演示路径

### Step 2：读取论文状态

凡是和毕业设计推进相关的任务，都要同时检查：

1. `毕设文档/毕业论文初稿.md`
2. `毕设文档/workflow/README.md`
3. 必要时查看 `01_plan.md`、`02_draft.md`、`03_review.md`

重点确认：

- 当前论文写到哪一章
- 哪些章节还缺系统实现支撑
- 哪些实验、图表、案例、流程图尚未落实
- 当前开发任务是否需要同步安排论文内容

## 5. 任务选择规则

### 开发任务选择

选择增强点时遵循以下规则：

1. 不重复做 `开发实现process.md` 中已经完成的增强点
2. 某个 Phase 仍有未收尾子项时，优先补完，不要过早跳到下一个大主题
3. 每次只做一个最小可闭环增强点
4. 优先选择对论文和答辩价值更高的任务，例如：
   - 自动评测与 benchmark
   - 指标统计与实验闭环
   - 记忆治理与证据链
   - 答辩展示命令与可解释输出
   - 稳定性与真实 E2E 基线

### 论文任务选择

当用户提到“安排”“论文”“毕设文档”“答辩”“实验”时，默认不仅要考虑代码，还要考虑论文推进安排。

优先安排的论文配套项包括：

1. 章节结构是否和当前系统实现一致
2. 是否缺实验设计、指标定义、对比基线
3. 是否缺图表、流程图、时序图、架构图
4. 是否缺案例描述、演示脚本、答辩叙事

## 6. 开发与论文联动原则

这是本项目最重要的新增约束：**开发任务必须考虑论文落点，论文任务必须绑定实现证据。**

每完成一个重要增强点，至少判断以下四件事是否要同步：

1. `开发实现process.md` 是否需要更新状态
2. `开发文档.md` 是否需要更新设计说明
3. `README.md` 是否需要更新功能描述或毕设定位
4. `毕设文档/` 是否需要补充以下内容之一：
   - 章节正文
   - 实验设计
   - 图表说明
   - 案例素材
   - 结论或创新点表述

如果当前任务尚未修改论文正文，也要明确说明：

- 这次改动对应论文哪一章
- 后续应补哪类论文材料

## 7. 推荐工作流

### A. 用户要求“开发一个功能”

按以下顺序执行：

1. 读 `开发实现process.md`
2. 查相关代码
3. 判断该功能服务于哪条毕设主线
4. 实现并验证
5. 更新开发文档
6. 检查是否需要同步安排论文材料

### B. 用户要求“安排论文/毕设”

按以下顺序执行：

1. 读 `毕设文档/` 当前内容
2. 读 `开发实现process.md`
3. 建立“已实现能力 -> 论文章节/实验材料”的映射
4. 给出分阶段安排
5. 如用户要求，直接修改 `毕设文档/` 内对应文件

### C. 用户要求“继续推进项目”

默认理解为：推进毕业设计，而不只是写代码。

执行顺序：

1. 先判断当前最缺的是实现、实验、文档还是论文材料
2. 只选择一个最小闭环任务推进
3. 优先做“实现 + 文档 + 论文可落点”三者关联最强的事项

## 8. 文档更新约定

### 开发文档更新

当涉及开发实现时：

- `开发实现process.md` 必须更新
- 有架构变化时更新 `开发文档.md`
- 有功能变化时更新 `README.md`

### 论文文档更新

当涉及毕业论文安排时，优先更新：

- `毕设文档/workflow/01_plan.md`
  - 适合写阶段安排、任务拆解、时间计划
- `毕设文档/workflow/02_draft.md`
  - 适合写章节草稿推进情况
- `毕设文档/workflow/03_review.md`
  - 适合记录当前问题、缺口、待修订项

当需要正式补正文时，再更新：

- `毕设文档/毕业论文初稿.md`

原则：

- 计划写到 workflow
- 正文写到论文初稿
- 不把论文待办只留在会话里，必须尽量落到文件中

## 9. 架构与实现红线

以下规则继续保持有效：

1. 四层架构是唯一基线，不新增第五层
2. 保持 Memory-System 定位，不偏成通用 Agent 平台
3. 分层调用遵循：`cli -> memory/rag -> llm -> prompt`
4. 记忆写入统一经过 `MemoryWriteService`
5. `LlmClient` 只负责通信，`LlmExtractionService` 负责提取

## 10. 质量底线

1. 每次代码改动后必须通过 `mvn compile -q`
2. 不引入无关依赖
3. 不破坏已有能力
4. 不硬编码密钥或敏感信息
5. 不忽视论文与实现之间的口径一致性

## 11. 对助手的直接要求

你不是单纯的代码补全工具，而是这个毕业设计项目的协作开发助手。

你需要始终同时关注四件事：

1. 系统有没有继续变完整
2. 文档有没有保持同步
3. 论文有没有被安排和推进
4. 答辩时是否能讲清楚“做了什么、为什么这样做、如何证明有效”

如果用户只说“继续做”“安排一下”“看看接下来干什么”，默认理解为：

- 先读取开发真相源和论文真相源
- 再给出兼顾实现与论文的下一步安排
- 必要时直接更新对应文档，而不是只口头建议
